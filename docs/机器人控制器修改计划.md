# 机器人控制器修改计划

## 目标

让 PLCCommunicator 回复给 pn_dev 的 80 字节状态帧包含真实机器人数据，而非全零。

## 当前问题

`buildRobotToPLCFrame()` 和 `notifyPlcActionDone()` 两个函数的代码都已写好，但从未被调用。
`robot_status_frame_` 始终是全零。TCP 收发正常，但数据内容是空的。

## 改动清单

| # | 文件 | 行号 | 类型 | 说明 |
|---|------|------|------|------|
| 1 | `robot_ros2.cpp` | 776 之后 | 新增 6 行 | 在 100Hz 定时器中刷新 PLC 状态帧 |
| 2 | `robot_ros2.cpp` | 818 之后 | 新增 1 行 | 在 jointStateCallback 中通知轨迹完成 |

---

### 改动 1：刷新 PLC 状态帧（解决"返回永远为 0"）

**文件**：`src/robot_bringup/src/robot_ros2.cpp`
**位置**：`updateRealtimeDataFromTimer()` 函数，第 776 行之后

**当前代码**（第 774-776 行）：
```cpp
    // 更新到 commander
    commander->updateRealTimeDataFromController(data);

    // static int update_count = 0;    // ← 在这行之前插入
```

**改为**：
```cpp
    // 更新到 commander
    commander->updateRealTimeDataFromController(data);

    // 同步刷新 PLC 状态帧（供 PLCCommunicator 回复给 pn_dev）
    if (plc_communicator_)
    {
        PLCFrame plc_frame = buildRobotToPLCFrame(data);
        plc_communicator_->updateRobotStatus(plc_frame);
    }

    // static int update_count = 0;
```

**为什么放在这里**：
- `updateRealtimeDataFromTimer()` 每 10ms 调用一次（100Hz）
- 它已经收集了完整的机器人状态（关节角度、IO、温度、模式等）
- 数据 `data` 就是 `RealTimeData`，正是 `buildRobotToPLCFrame()` 需要的参数
- 更新频率 100Hz 对 PROFINET 1ms 周期来说略微滞后，但状态监测足够

---

### 改动 2：让 BOOL[16] 动作完成信号生效

**文件**：`src/robot_bringup/src/robot_ros2.cpp`
**位置**：`jointStateCallback()` 函数末尾，第 819 行（`}` 之前）

**当前代码**（第 818-820 行）：
```cpp
            if (src_idx < msg->effort.size())
                latest_joint_state_.effort[i] = msg->effort[src_idx];
        }
    }
}
```

**改为**：
```cpp
            if (src_idx < msg->effort.size())
                latest_joint_state_.effort[i] = msg->effort[src_idx];
        }
    }

    // 通知 PLC 轨迹完成（检测 is_active 下降沿，置位 BOOL[16] 保持 300ms）
    notifyPlcActionDone(trajectory_cache_.is_active);
}
```

**为什么放在这里**：
- `jointStateCallback` 是 1kHz 的关节状态回调
- 函数注释（第 5152 行）明确写了"在 jointStateCallback 末尾调用"
- `notifyPlcActionDone()` 检测轨迹 `is_active` 的下降沿，置位 BOOL[16] 保持 300ms
- 这个函数是"纯状态观测，不影响运动执行"，放在回调末尾安全

## 不改的部分

以下问题**本次不修改**：

| 问题 | 原因 |
|------|------|
| `disableMotors()` 是 TODO 桩 | 假设急停通过其他通道（EtherCAT SDO）处理，暂不验证 |
| TCP recv 无超时 | 因为 pn_dev 每 1ms 发一次，recv 不会真正永久阻塞 |
| `listen(server_fd, 1)` 积压=1 | 只有 pn_dev 一个客户端，足够了 |
| 第一帧边缘误判 | 只要 PLC 端初始值全为 0，就不会误触发 |

## 改动后的效果

```
修改前：
  pn_dev send(PLCFrame) → PLCCommunicator recv → handlePLCData ✓
                                                    ↓
                         PLCCommunicator send(robot_status_frame_) → 全是 0
                                                    ↑
                                            没人更新这个变量

修改后：
  pn_dev send(PLCFrame) → PLCCommunicator recv → handlePLCData ✓
                                                    ↓
                         PLCCommunicator send(robot_status_frame_) → 真实数据
                                                    ↑
                                           100Hz 定时器持续刷新
                                           buildRobotToPLCFrame() → updateRobotStatus()
```

## 验证步骤

1. 编译运动控制器：`colcon build`
2. 启动运动控制器，确认 31000 端口监听
3. 启动 pn_dev，确认 "TCP bridge: connected"
4. 用 `pn_shm_tool` 或 PLC 模拟器发包到 31000，检查回复的 80 字节：
   - `BOOL[0]` = 电源状态
   - `BOOL[2]` = 已使能
   - `BOOL[6]` = Busy
   - `BOOL[7]` = Moving
   等等不应全为 0
